# Memory — Контекст проєкту Flowi для Claude

Ця папка містить **довгостроковий контекст** про додаток, який Claude може
підвантажити на початку нової сесії, щоб не починати з нуля.

## Структура

| Файл | Що містить | Коли читати |
|------|------------|-------------|
| `README.md` | Цей файл — мапа |  завжди |
| `architecture.md` | Структура папок, технологічний стек, ключові патерни | при будь-якій задачі, що змінює >1 файл |
| `screens.md` | Огляд кожного екрана: призначення, ключі storage, особливості | при роботі з конкретним екраном |
| `conventions.md` | Правила коду, кольори, стилі, заборонені практики | завжди перед написанням коду |
| `server-api.md` | Django ендпоінти і WebSocket-канали | при змінах на сервері або синхронізації |
| `data-model.md` | Типи даних, AsyncStorage ключі, формати | при змінах структури даних |
| `changelog.md` | Хронологія сесій | для contextual awareness |

## Як використовувати

Claude має:
1. На старті сесії прочитати `README.md` (цей файл).
2. Прочитати конкретний файл за темою задачі (з таблиці вище).
3. Не дублювати інформацію, яку легше отримати з git (історія комітів,
   `git blame`). Memory — про **рішення, домовленості, причини**.

## Що НЕ зберігати

- Тимчасові стани й TODO — для них `TaskCreate` під час сесії.
- Дослівні дамп коду — він є в репозиторії.
- Імена змінних, що часто переіменовуються — крихко.
- Коментарі-обгортки до того, що очевидно з коду.

## Що варто зберігати

- **Чому** так зроблено (рішення з компромісами).
- **Заборони** з причиною (вже наступали на граблі).
- **Не-очевидні залежності** між модулями.
- **Інтеграційні точки** з зовнішніми сервісами.

## Оновлення

Після значущих змін у проєкті (нова фіча, рефакторинг архітектури):
- `changelog.md` — додати рядок з датою і резюме.
- `architecture.md` / `screens.md` / тощо — оновити лише якщо щось зламається без цього знання в наступній сесії.
