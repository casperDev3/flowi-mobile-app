# План редизайну екрана «Здоров'я» — хаб + окремі модулі

**Дата:** 2026-06-29
**Мета:** перетворити суцільний скрол-екран на **дашборд зі зведеною статистикою** + сітку **плиток-розділів**, кожна з яких відкриває окремий Stack-екран із власним функціоналом.

> ✅ **СТАТУС: реалізовано (Фази 0–3).** Хаб + 6 розділів + профілактика (4 під-екрани з нотифікаціями) + експорт звіту. Деталі — у кінці документа («Підсумок реалізації»).

**Рішення (узгоджено):**
- Розділи: **Тренування**, **Харчування**, **Активність**, **Сон і відновлення**, **Показники тіла**, **Профілактика** (+ Профіль у хедері).
- «Профілактика» = **Ліки/добавки** + **Медогляди/аналізи** + **Щеплення** + **Звички/чек-лісти**.
- Навігація — **окремі Stack-екрани** (`app/health-*.tsx`).

## 1. Аналіз поточних процесів (як є зараз)

### 1.1. Поточна структура екрана
`app/(tabs)/health.tsx` (~970 рядків) — один великий екран зі секціями: overview rings → quick stats (вага/ІМТ/пульс) → калорії+чіпи → білок → кроки → вода → сон → вага → пульс → нагадування. Плюс банер «Тренування» (→ `/workouts`), модалка вводу, модалка історії, радіальний FAB, кнопки профілю/історії в хедері.

### 1.2. Потоки даних (як працює запис/читання)
```
AsyncStorage 'health_entries_v2'  ──load/migrate──▶  state entries: HealthEntry[]
        ▲                                                    │
        │ saveData (useEffect on entries)                    ├─ sumForDay()  → todayWater/CalIn/CalOut/Steps
        │                                                    ├─ lastForDay() → todayWeight/Sleep/Pulse
HealthKit (iOS) ──fetchTodayData()──▶ merge (note '__hk__')  ├─ chartData() → 7-денні чарти
                                                             └─ computeGoals(profile, weight) → цілі
'health_profile'  ──load──▶ profile ─────────────────────────┘
'health_reminders' ──load──▶ Switch-и нагадувань → scheduleDailyReminder()
```

**Типи записів (один масив, дискримінатор `type`):**
`water · calories · calories_out · steps · weight · sleep · pulse` (+ опц. `protein/fat/carbs` на food-записах).

**Ключова ідея для редизайну:** усе вже лежить в **одному масиві** `HealthEntry[]`, агрегати — чисті функції від `(entries, day/type)`. Тому поділ на модулі — це **поділ представлення**, а не даних: кожен модуль читає той самий масив, фільтруючи потрібні `type`. Дані й синхронізація не ламаються.

### 1.3. Проблеми поточного підходу
- Один файл ~970 рядків тримає весь стан + усі секції + 2 модалки → важко розширювати.
- Логіка вводу (модалка add) і агрегати продубльовані інлайн — їх не переюзати з інших екранів.
- Немає «верхнього рівня» — користувач одразу падає у вертикальний потік усіх метрик.
- «Профілактика» (ліки/огляди/щеплення) повністю відсутня.

## 3. Модель даних

### 3.1. Існуючі дані — без змін
`health_entries_v2: HealthEntry[]`, `health_profile`, `health_reminders` лишаються. Модулі Харчування/Активність/Сон/Показники читають той самий масив за `type`. Жодної міграції.

| Модуль | Читає типи | Пише типи |
|--------|-----------|-----------|
| Харчування | `calories`, `calories_out`, `water` (+macros) | `calories`, `water` |
| Активність | `steps`, `calories_out` | `steps` |
| Сон | `sleep`, `pulse` (спокою) | `sleep` |
| Показники тіла | `weight`, `pulse` | `weight`, `pulse` |

### 3.2. Нові дані — модуль «Профілактика» (масиви з `id` → сумісні зі sync)
```ts
// store key 'health_meds'
interface Medication {
  id: string; name: string; dose?: string;       // "500 мг", "2 капсули"
  times: string[];          // ["08:00","20:00"] — час прийому
  days?: number[];          // дні тижня 0..6; порожньо = щодня
  startDate: string; endDate?: string;            // курс
  active: boolean; createdAt: string;
  log?: { date: string; takenAt: string }[];      // відмітки прийому
  notifIds?: string[];                            // id запланованих нотифікацій
}

// store key 'health_checkups'
interface Checkup {
  id: string; kind: 'analysis' | 'visit' | 'procedure';
  title: string; date: string; result?: string; notes?: string;
  nextDate?: string;                              // нагадати про наступний
  createdAt: string; notifId?: string;
}

// store key 'health_vaccines'
interface Vaccine {
  id: string; name: string; date: string; doseNo?: number;
  nextDate?: string; notes?: string; createdAt: string; notifId?: string;
}

// store key 'health_habits'
interface Habit {
  id: string; title: string; icon: string; color: string;
  schedule: 'daily' | 'weekly'; targetPerDay?: number;
  log: string[];                                  // ISO-дати виконання
  streak?: number; createdAt: string; reminderAt?: string; notifId?: string;
}
```

### 3.3. Нагадування
- **Ліки** — кілька разів/день → планувати N нотифікацій на запис (`times[]`). Потрібен **повторюваний** тригер (DAILY) на кожен час. Розширити `notifications.ts`: `scheduleMedReminders(med)` / `cancelMedReminders(notifIds)`.
- **Огляди/щеплення** — разові DATE-нотифікації на `nextDate` (реюз наявного `scheduleReminder`-патерну, узагальнити під довільний id).
- **Звички** — щоденний DAILY (реюз `scheduleDailyReminder`).

## 4. Деталізація екранів

### 4.1. ХАБ `(tabs)/health.tsx` (переписати, ~250 рядків)
- Хедер: заголовок + кнопка профілю + кнопка історії.
- **Зведена статистика**: `RingCell ×4` (калорії-баланс, кроки, вода, сон) + рядок vitals (вага/ІМТ/пульс) + опц. інсайт тижня.
- **Сітка плиток** (`HubTile ×6`) з live-показниками з `useHealthEntries`/нових ключів.
- Радіальний FAB швидкого вводу лишається (через `HealthEntryModal`).
- Дані: `useHealthEntries()` + лічильники Профілактики (кількість «на сьогодні»: невипиті ліки, заплановані огляди).

### 4.2. Харчування `/health-nutrition`
Картки: `CalorieCard` (спожито/спалено/баланс, ліміт) + `ProteinCard` + `WaterCard` + (опц.) розклад БЖВ кільцями. Чіпи швидкого додавання + FAB. Журнал їжі за день (записи `calories` з нотатками/макросами).

### 4.3. Активність `/health-activity`
`StepsCard` (кроки, дистанція від зросту, 7-денний чарт, % мети) + активні калорії (`calories_out`) + (майб.) HR-зони тренування. Чіпи кроків + FAB.

### 4.4. Сон і відновлення `/health-sleep`
`SleepCard` (тривалість, якість, прогрес до цілі, 7-денний чарт) + пульс спокою (resting HR з HealthKit) + поради. Ввід сну.

### 4.5. Показники тіла `/health-vitals`
`WeightCard` (вага, ІМТ-бейдж, тренд кг/тиж, чарт + ціль ваги) + `PulseCard` (зони ЧСС) + **тиск** (новий тип `bp` — систолічний/діастолічний, опційно). Ввід ваги/пульсу/тиску.

### 4.6. Профілактика `/health-prevention` (суб-хаб)
4 плитки → окремі екрани:
- **Ліки** `/health-meds`: список курсів, час прийому, відмітка «прийняв», % дотримання, нотифікації.
- **Медогляди** `/health-checkups`: список оглядів/аналізів, дати, результати, нагадування про наступний.
- **Щеплення** `/health-vaccines`: журнал щеплень, дози, наступна дата.
- **Звички** `/health-habits`: щоденні чек-лісти + серії (streaks).

## 7. Ризики й сумісність
- **Зворотна сумісність даних:** повна — існуючі записи/профіль/нагадування не чіпаються; модулі читають той самий масив. Міграції не потрібно (крім уже наявної `calories→calories_out`).
- **Sync нових ключів:** масиви з `id` сумісні з `mergeArraysWithConflicts`; додати у `SYNC_KEYS`/`data.tsx` обережно (об'єкти-профіль НЕ додавати у sync).
- **Нотифікації-ліки:** ліміт запланованих нотифікацій на iOS (64) — кешувати лише активні курси, чистити при деактивації.
- **Обсяг файлів:** Фаза 0 критична — без винесення хука/карток модулі дублюватимуть логіку. Робити її першою.
- **HealthKit:** синк лишається на рівні хука (один раз), модулі лише читають результат.

- Модулі читають той самий `health_entries_v2` через хук (нуль міграцій). Зміни синхронізуються між екранами через `useFocusEffect`-reload.
- Профілактика: ліки (час прийому, відмітка, % дотримання, щоденні нотифікації), медогляди/щеплення (разові DATE-нагадування), звички (серії + щоденні нагадування), **експорт звіту для лікаря** через `Share`.

**Свідомі рішення/обмеження:**
- Дати в формах профілактики — текстом `YYYY-MM-DD` (немає date-picker у залежностях). Час — `HH:MM`.
- Звіт лікарю — текстовий `Share.share` (не PDF; `expo-print` не встановлено).
- `health_profile`/`health_reminders` лишаються локальними (об'єкти; sync мерджить лише масиви). 4 масиви профілактики — синхронізуються.
- Кожен екран має власний екземпляр хука → повторний HealthKit-синк при відкритті модуля (прийнятно; кандидат на винесення у глобальний контекст).

**Перевірки:** усі 109 `tr.*`-ключів присутні; усі зареєстровані маршрути мають файли; дублікати i18n-ключів усунено. **Типи/лінт локально не прогнано** (`node_modules` відсутні) — радимо `npx tsc --noEmit` + `expo lint` на машині розробника.

**Наступні кроки (не входили в цей етап):** тижневі інсайти на хабі, HR-зони Z1–Z5 + resting HR з HealthKit, тиск, ціль ваги з графіком 3 міс, винесення HK-синку в глобальний контекст, date-picker та PDF-звіт.
