## Частина A. «Спільне» (`shared.tsx`)

Колаборативні групи: списки **Покупки / Завдання / Нотатки**, синхронізація через бекенд (`flowi-server-app.vercel.app`), офлайн-first з локальним кешем, приєднання за секрет-кодом, WebSocket для realtime.

### A.1. UI/UX — знахідки

| # | Проблема | Тяжкість | Деталі |
|---|----------|----------|--------|
| U1 | **FAB перекриває таб-бар** | 🔴 | `bottom: iOS 48 / 28` — це значення для Stack-скрінів. «Спільне» — **вкладка**, FAB має бути `108 / 88` (правило CLAUDE.md), інакше «+» налазить на таб-бар |
| U2 | **Хардкод-рядки, нема EN** | 🟠 | 'Покупки', 'Додати', 'Створити групу' тощо напряму, не через `tr.*` — порушує i18n-правило; англійська версія відсутня |
| U3 | **Нотатки з чекбоксами** | 🟠 | Секція «Нотатки» використовує той самий чекбокс + закреслення, що й покупки — невірна метафора; нотатки не «виконуються» |
| U4 | **Секрет не скопіювати/поділитися** | 🟠 | Код показано, але нема кнопок «Копіювати»/«Поділитися»/QR — користувач переписує вручну (висока тертя для головного flow віральності) |
| U5 | **Немає undo при видаленні** | 🟠 | Видалення одразу (soft-delete), без «Скасувати» — легко втратити пункт |
| U6 | **«Очистити куплені» без підтвердження** | 🟡 | Масово видаляє + N мережевих запитів у циклі, без діалогу |
| U7 | **Тихі помилки синхронізації** | 🟡 | `sync failed` лише в `__DEV__`; користувач не бачить, що дані не зберігаються на сервер |
| U8 | **Немає списку учасників/присутності** | 🟡 | Лише лічильник; не видно, хто в групі та хто онлайн |
| U9 | **Немає атрибуції пунктів** | 🟡 | Не видно, хто додав/змінив пункт — слабко для спільної роботи |
| U10 | **Початкове завантаження без скелета** | 🟡 | `initialized` не показує лоадер; коротке «блимання» порожнім станом |

### A.2. Технічні ризики (впливають на UX)

- 🔴 **`crypto.randomUUID()`** (рядки 138, 318) — глобальний `crypto` може бути відсутнім у Hermes → краш при першому запуску/додаванні. Потрібен `expo-crypto` або `react-native-get-random-values`.
- 🔴 **Realtime через Vercel** — `wss://…vercel.app/ws/...` майже напевно не тримається (serverless). Realtime-оновлення не працюватимуть; лишається лише sync-on-focus/pull.
- 🟠 **Втрата правок при merge** — `applyRemoteItem` завжди перезаписує локальний пункт віддаленим без порівняння `updated_at` → одночасні правки можуть затирати свіжіші локальні.
- 🟠 **`syncWithServer` читає `items` із замикання** — можливе використання застарілого стану при відправці.

### A.3. Бізнес-аналітика

- **Цінність:** колаборація — головний драйвер **віральності** (запросив члена сім'ї → новий користувач) та **утримання** (спільний список = щоденне повернення). Зараз потенціал недовикористаний: нема легкого запрошення (copy/QR/share), присутності, атрибуції.
- **Воронка приєднання** має зайве тертя (ручне введення коду) — головна точка втрат для віральної петлі.
- **Монетизація:** спільні групи — природний premium (необмежені групи/учасники/історія; приватність). Безкоштовно — 1 група, до N учасників.
- **Ризик залежності:** функція повністю зав'язана на зовнішній сервер; його простій = непрацездатність. Потрібні чіткі офлайн-стани та індикатор з'єднання.

### A.4. Рекомендації «Спільне»

1. Перенести FAB на таб-висоту (`108/88`).
2. i18n усіх рядків (uk+en).
3. Нотатки — без чекбоксів (текстові картки, багаторядкові).
4. Кнопки **Копіювати** + **Поділитися** + **QR** для секрету (різко знижує тертя приєднання).
5. Undo-тост при видаленні; підтвердження «Очистити куплені».
6. Видимий індикатор синку/помилки (банер «Не вдалося синхронізувати»).
7. Conflict-resolution за `updated_at` (last-write-wins з порівнянням, не сліпий overwrite).
8. Полагодити/замінити realtime або чесно прибрати WS і покладатися на polling.
9. Список учасників + (опц.) атрибуція пунктів.

## Частина B. «Здоров'я» (хаб + модулі)

Нова архітектура: хаб зі зведеною статистикою + плитки → окремі екрани (Харчування/Активність/Сон/Показники/Тренування/Профілактика/Профіль). Профілактика — нижній лист. Деталі попереднього аудиту — у `reviews/HEALTH_SCREEN_REVIEW.md`.

### B.1. UI/UX — знахідки

| # | Проблема | Тяжкість | Деталі |
|---|----------|----------|--------|
| H1 | **Дати/час — текстові поля** | 🟠 | Профілактика: дата `YYYY-MM-DD`, час `HH:MM` вручну — висока ймовірність помилок вводу; потрібні нативні picker'и |
| H2 | **Повторний HealthKit-синк** | 🟠 | Кожен модуль має свій `useHealthEntries` → HK init/sync при відкритті кожного екрана; варто винести в глобальний контекст/провайдер |
| H3 | **Звіт лікарю — лише текст** | 🟡 | `Share.share({message})`; нема PDF — менш «офіційний» вигляд для лікаря |
| H4 | **Немає тижневих інсайтів** | 🟡 | Хаб має місце під інсайт («Сон ↓12%»), але поки не реалізовано — втрата приводу повертатись |
| H5 | **Профіль може бути не заповнений** | 🟡 | Цілі тоді на `FALLBACK_WEIGHT` — числа можуть вводити в оману без онбордингу |
| H6 | **Тиск/ширші vitals відкладено** | 🟡 | Лише вага/ІМТ/пульс; немає тиску, % жиру, обводів |

### B.2. Бізнес-аналітика

- **Цінність:** здоров'я — ядро щоденної залученості; **нагадування + серії (streaks) + профілактика** прямо працюють на DAU та утримання.
- **Диференціація:** не дублювати Apple Health, а додавати інтерпретацію (цілі, інсайти, профілактика, звіт лікарю).
- **Монетизація:** Premium — макро-аналітика, інсайти/тренди, PDF-звіт лікарю, синхронізація профілю між пристроями, AI-поради (Claude API).
- **Зачіпки утримання:** щоденні нагадування (вода/сон/ліки/звички), серії звичок, нагадування про огляди/щеплення — усі вже закладені, треба довести інсайти.

### B.3. Рекомендації «Здоров'я»

1. Нативні date/time picker'и у профілактиці (UX-критично для дат).
2. Винести `useHealthEntries` у провайдер (один HK-синк, спільний стан).
3. Тижневі інсайти на хабі (дельти 7 днів) — реалізувати закладене місце.
4. Онбординг-нагадування заповнити профіль (банер уже є — підсилити).
5. PDF-звіт (`expo-print`) як преміум-апгрейд текстового.
